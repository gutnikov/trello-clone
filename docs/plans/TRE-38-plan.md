# Implementation Plan: TRE-38

## Overview

Implement the card API router with four endpoints (create, update title, delete, move/reorder) in a new `cards.py` module, register it in `main.py`, and write comprehensive API tests. The database layer already provides all necessary CRUD methods (`create_card`, `get_card`, `update_card`, `delete_card`, `get_cards_by_list`) and the `Card` model is defined. This is the first router module in the project, so it establishes the pattern for boards and lists routers. No existing routers exist to copy from, but the DB layer patterns and test fixtures (`db`, `client` in `conftest.py`) are ready to use.

### Step 5: Implement PUT /cards/{id}/move endpoint (move/reorder)

- **Files:** `backend/src/app/routers/cards.py` (modify)
- **Changes:**
  - Add `@router.put("/cards/{card_id}/move")` handler.
  - Accept `card_id: str` path parameter, `body: MoveCardRequest`, and `request: Request`.
  - Get DB via `_get_db(request)`.
  - Fetch the current card via `await db.get_card(card_id)`.
  - If card is `None`, raise `HTTPException(status_code=404, detail="Card not found")`.
  - Determine source list (`card.list_id`) and target list (`body.list_id`).
  - **Reorder logic:**
    1. Get all cards in the source list via `db.get_cards_by_list(card.list_id)`.
    2. Remove the moving card from the source list's card list.
    3. Update positions of remaining source list cards sequentially (0, 1, 2, ...).
    4. If target list differs from source list, get all cards in the target list via `db.get_cards_by_list(body.list_id)`.
    5. Insert the moving card at the requested `body.position` in the target list's card list.
    6. Update positions of all target list cards sequentially (0, 1, 2, ...).
    7. If target list equals source list (reorder within same list), work with a single combined operation: remove card, insert at new position, reassign all positions.
    8. Use `db.update_card(card.id, list_id=body.list_id, position=body.position)` to persist the move.
    9. For each displaced card in source and target lists, call `db.update_card(c.id, position=new_pos)`.
  - Return the updated card via `updated.model_dump()`.
  - Log `card_moved` with `card_id`, `source_list_id`, `target_list_id`, `new_position`.
- **Depends on:** Step 1

## Risk Factors

- **First router in the project** — No existing router pattern to copy. The implementation establishes the convention for all future routers (boards, lists). Mitigation: keep the pattern simple and consistent with FastAPI conventions. Use `request.app.state.db` for DB access (matching the established pattern in `conftest.py:57`). Future routers should copy this pattern.

- **Move/reorder position management complexity** — The move endpoint must update positions of multiple cards across potentially two lists. Race conditions are possible if multiple concurrent moves happen. Mitigation: SQLite uses file-level locking, so concurrent writes are serialized. For correctness, fetch current state, compute new positions, and update all affected cards within a single handler call. Consider wrapping the multi-card updates in a transaction (the DB class commits after each `update_card` call, which means intermediate states are visible). The implementation should call `db.update_card()` for each affected card and then commit. Since `Database.update_card` calls `commit()` after each update, intermediate states exist briefly but are acceptable for v0.

- **Position auto-assignment relies on list length** — Using `len(existing_cards)` assumes positions are 0-indexed and contiguous. If cards are deleted, gaps may exist but the count still gives the next available position. This is correct behavior — new cards go to the end regardless of gaps.

- **No validation that list_id exists** — The POST create and PUT move endpoints accept a `list_id` but don't verify the list exists before creating/moving the card. SQLite foreign key constraints will reject the insert/update if the list doesn't exist, but the error will be a 500 (sqlite IntegrityError) rather than a 400. Mitigation: acceptable for v0; a future issue can add explicit list existence validation with proper 400/422 error responses.

- **Request schema vs domain model confusion** — The `Card` model is frozen (immutable), so request bodies must use separate Pydantic models (not `Card` directly). The plan uses `CreateCardRequest`, `UpdateCardRequest`, and `MoveCardRequest` schemas. This is correct and avoids the pitfall of trying to use frozen models for mutable request parsing.

- **Type annotations for `request.app.state.db`** — Starlette's `app.state` is `Any`, so accessing `.db` won't be type-checked by mypy. The `_get_db` helper provides a typed return value, mitigating mypy strictness issues. May need `# type: ignore[no-any-return]` on the return statement.
